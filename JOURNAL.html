Chapter II — The Five Alternatives

Catalogued five orthogonal directions to make the simulation faster & more capable. Tackled in order, with honest benchmarking.
WhenKindEntry
Alt #1 Build Closed-form equation for instant restoration-time estimate. Powers the "≈ 4 d 3 h" live readout that updates as the slider moves. Useful for slider-sweeping; the exact answer still comes from Plan restoration.
Alt #2 Build Pre-computed scenario library. 12 named scenarios saved as JSON in scenarios/, loadable from a dropdown. ~12.5 MB total. Lets viewers explore without waiting for a fresh simulation.
Alt #3 Build WebAssembly scheduler in Rust (#![no_std], custom inline Taylor-series math, bump allocator, parallel-array heap, spatial grid hash). Compiled artifact at wasm/scheduler.wasm.
Alt #3 Honest result Benchmark: WASM consistently 2–3× slower than V8 JS at 500–2 000 outages. Root cause: hand-rolled Taylor-series math is no match for V8's native intrinsics. Conclusion — the JS scheduler already wins. WASM stays as a reference build, not the production path.
Alt #4 Question "move onto #4"
Alt #4 Build FastAPI backend (07_server.py) with /api/schedule and /api/monte_carlo. Wide-open CORS so GitHub Pages can call it. Dockerfile for deployment.

Chapter III — The Scaling Saga

Where a 25 000-outage × 5 000-crew "max settings" stress test ran for eight minutes — and then half a second.
WhenKindEntry
Server v0 Question "its taking forever to load … 25k outages × 5 000 crews"
Step 1 +5× NumPy vectorisation: inner loop uses np.argmin over a masked distance array; np.searchsorted on a sorted discovery list replaces the O(N) "next-discovery" scan. 2 k × 100 went from 0.34 s → ~0.05 s.
Step 2 Choice User: "do B" (Option B = NumPy port). KD-tree was tried (#1) and proved slower in realistic mode (discovered-fraction is low early on, K grows). Capped K to 256, useful only at non-realistic dense regimes.
Step 3 +14× Numba JIT of the entire dispatch loop: inline binary heap, inline haversine, inline Mulberry32, @njit(cache=True, fastmath=True). 2 k × 100 → 24 ms. Process pool tried for Monte Carlo but Windows spawn overhead beat it; reserved for huge jobs only.
Step 4 Pivot User: "make the model survive even unrealistic scenarios … going to scale to Connecticut." Skipped MILP/calibration; focused on raw scalability.
Step 5 +246× Spatial grid hash (G = √(N/5) cells, Chebyshev ring expansion) plus an incrementally-maintained n_available counter. The counter is the real hero: when no work is discovered yet, crews fast-forward to the next discovery via binary search instead of doing a full grid scan over an empty layer.

Chapter IV — Deployment & Polishing

From localhost to a public Render URL, with a four-failure deploy saga along the way.
WhenKindEntry
Deploy Build Render Blueprint (render.yaml) auto-provisions the backend on push. Dockerfile pins fastapi / uvicorn / numpy / scipy / numba.
Deploy Fix First failure: matplotlib + numpy required. Cause: 05_generate_artifacts.py imports matplotlib unconditionally, and 07_server.py imported it via importlib. Added matplotlib to image.
Deploy Fix Second failure: FileNotFoundError: data/hartford_boundary.json — same file tries to load boundary data at module level. Wrapped in try/except Exception. Still failed.
Deploy Fix (real) Third failure: same error. Reason: 05_generate_artifacts.py raises SystemExit, which inherits from BaseException, not Exception — Python deliberately makes except Exception skip it. Stopped importing 05_generate_artifacts entirely; fast paths cover everything. Classic gotcha — and a good reminder that Python's exception hierarchy is deliberate.
Frontend Build Auto-warm + auto-detect: page load fires a no-cors GET to wake the Render container; /version probe at 5 s / 30 s / 60 s; health dot in the panel goes gray → amber → green. Plan restoration silently falls back to in-browser when server is offline.
Frontend Build Server progress bar — exponential asymptote toward 90 % while waiting on a single atomic server call, snaps to 100 % on response. Designed so Render cold starts (~30–60 s) still feel like motion.
Frontend Question "why did you stop with those images that we used to have for when the plan restoration works"
Frontend Build Flat per-job dispatch log added inside Numba scheduler; server returns full jobs[] per crew. Browser renderPlan() extracted so both local and server paths produce the same colored crew depots + numbered repair circles + canvas point cloud.

Chapter V — Conversations & Direction

The questions that steered the work.
WhenKindEntry
Mid-saga Question "MLP/calibration is what I want to do if it helps leads me towards more publishable research contributions"
Mid-saga Reframe Pitched the trade: MILP optimal scheduler (1 day; needs no external data) vs. Eversource calibration (1 day code but bottlenecked on data access via PURA dockets). Recommended doing MILP first so the calibration plugs into a ready framework.
Mid-saga Question "the real storm data is a lot later. We need to make this more scalable and more precise for Hartford county first … going to need to handle even heavier scenarios when scaled up to connecticut"
Mid-saga Pivot Skipped MILP for now. Refocused on raw scalability so the model can survive CT-wide projection. Result: the Chapter III speedups above.
Now Decision Next research milestones queued, not yet built:
  • MILP optimal scheduler for small N to bound greedy's optimality gap.
  • Eversource / PURA storm calibration once data acquired.
  • WebGPU as engineering polish — deferred; Numba already handles CT scale.
